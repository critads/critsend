TASK: Fix Neon “database connection error” under high load (imports + parallel jobs) in our Replit TypeScript (Express) SaaS.

CONTEXT:
We use Neon Postgres. During big imports or multiple jobs in parallel we often get “database connection error”. This is almost always caused by too many concurrent connections / missing pooling / creating pools multiple times. We must switch to Neon pooled connection and enforce backpressure + batching.

GOAL:
1) Stop connection errors during large imports and parallel workers.
2) Use ONE shared DB pool per process (API server, worker, importer).
3) Limit concurrency and batch DB writes/reads.
4) Keep changes minimal and safe.

DO THIS STEP-BY-STEP:

1) Ensure we use Neon POOLED connection string
- In Neon dashboard, select the connection string labeled “Pooled” (host contains “-pooler”).
- Update Replit Secrets:
  DATABASE_URL = <NEON POOLED URL> (must include sslmode=require)
  PG_POOL_MAX = 5
- Add a check at startup: if DATABASE_URL does NOT include “-pooler”, log an error and refuse to start.

2) Enforce a SINGLETON Postgres pool (no new Pool per request/job)
- Create src/db.ts (or server/db.ts) with exactly ONE Pool instance exported.
- IMPORTANT: Search the whole repo for any “new Pool(” or “new Client(” (or ORM clients that open connections) and refactor them to reuse the singleton pool.
- The pool must NOT be created inside request handlers, job processors, import scripts, or utility functions.

Use this exact file:

--- src/db.ts ---
import { Pool } from "pg";

if (!process.env.DATABASE_URL) throw new Error("Missing DATABASE_URL");
if (!process.env.DATABASE_URL.includes("-pooler")) {
  throw new Error("DATABASE_URL must be the Neon POOLED URL (host contains -pooler).");
}

export const pool = new Pool({
  connectionString: process.env.DATABASE_URL,
  ssl: { rejectUnauthorized: false },
  max: Number(process.env.PG_POOL_MAX ?? 5),     // KEEP SMALL
  idleTimeoutMillis: 30_000,
  connectionTimeoutMillis: 10_000,
});
-----------------

Then update all DB code to import { pool } from "./db".

3) Add graceful shutdown (prevents leaked connections on redeploy)
- In the backend entrypoint (server.ts / index.ts), add:

import { pool } from "./db";

async function shutdown() {
  try { await pool.end(); } catch {}
  process.exit(0);
}
process.on("SIGTERM", shutdown);
process.on("SIGINT", shutdown);

4) Cap parallel DB usage (backpressure)
- Identify places where we do Promise.all over large arrays with DB queries, and replace with concurrency-limited execution.
- Install p-limit and use it where we do mass inserts/updates/reads:

npm i p-limit

Example pattern:
import pLimit from "p-limit";
const limit = pLimit(10);
await Promise.all(items.map(x => limit(() => dbOp(x))));

- For workers (BullMQ or any job runner), set worker concurrency LOW:
  imports: concurrency 1–3
  send jobs: concurrency 10–20 (start low)
- Remove any unbounded parallel imports.

5) Fix big imports: BATCH + reuse one client
- Imports must run in chunks of 5,000–20,000 rows per batch.
- Use a single connection per import process (client = await pool.connect()) and reuse it through the whole import.
- Commit per batch (BEGIN/COMMIT). If error: ROLLBACK and continue/stop safely.
- Do NOT open a new connection per batch/row.

Provide an importer pattern like:

const client = await pool.connect();
try {
  for (const batch of batches) {
    await client.query("BEGIN");
    // batched upserts/inserts here
    await client.query("COMMIT");
  }
} catch (e) {
  await client.query("ROLLBACK");
  throw e;
} finally {
  client.release();
}

6) Add visibility to confirm fix
- Add periodic logs of pool stats:
setInterval(() => {
  // @ts-ignore
  console.log("PG pool stats", { total: pool.totalCount, idle: pool.idleCount, waiting: pool.waitingCount });
}, 10_000);

If waitingCount grows, reduce concurrency further.

ACCEPTANCE CRITERIA:
- Large import no longer triggers “database connection error”.
- Running multiple jobs in parallel no longer breaks DB connectivity.
- Codebase has ONLY ONE pool creation per process (no “new Pool” elsewhere).
- Pool waitingCount remains bounded during load tests.
