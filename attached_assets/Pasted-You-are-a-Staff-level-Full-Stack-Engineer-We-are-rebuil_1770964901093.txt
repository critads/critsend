CONTEXT / PRODUCT REQUIREMENTS
Segments allow users to target a subset of contacts (subscribers/leads) for campaigns. A segment is defined by rules/filters on contact attributes and events. We need:
1) Segment Builder (UI) with nested logic:
   - Groups with AND/OR
   - Nested groups
   - Conditions like:
     - Contact fields: email, phone, firstName, lastName, gender, country, city, language, tags, source, listId, createdAt, updatedAt
     - Engagement: opened emails, clicked links, lastOpenAt, lastClickAt, numberOfOpens, numberOfClicks
     - Suppression states: unsubscribed, bounced, complained
     - Custom fields (dynamic schema per tenant)
2) Operators:
   - equals, notEquals
   - contains, notContains
   - startsWith, endsWith
   - in / notIn (lists)
   - exists / notExists
   - greaterThan, greaterOrEqual, lessThan, lessOrEqual
   - between (for dates/numbers)
   - isTrue / isFalse
3) Segment types:
   - “Dynamic segment” (computed from rules)
   - “Static snapshot” (optional): ability to freeze results at a point in time (can be phase 2, but design should allow it cleanly)
4) Preview:
   - Show estimated count + preview first N contacts
   - Must be fast and resilient for large datasets
5) Save / Edit / Duplicate / Delete segments
6) Use in campaigns:
   - API returns list of contact IDs (paginated) for sending jobs
   - Must handle large segments without loading everything into memory

NON-FUNCTIONAL REQUIREMENTS (CRITICAL)
- Multi-tenant safety: every query MUST be scoped by tenantId/accountId.
- Correctness first: nested logic must be reliably represented and evaluated.
- Performance: handle millions of contacts, avoid full table scans, use indexes, pagination, and query planning.
- Security: strict validation, prevent injection, never allow raw SQL from client.
- Reliability: no random “database connection error” under load. Limit concurrency, use pooling properly, and ensure query composition is efficient.
- Observability: structured logs, meaningful error messages, tracing-like correlation id, and metrics hooks.
- Testing: unit tests for rule-to-query compiler + integration tests for API.
- DX: clean types, clear folder structure, minimal tech debt.

Step 3 — Query Compiler (Rules -> SQL)

- Absolutely no string concatenation for values. Parameterize everything.
- Whitelist fields/operators (explicit mapping).
- Support joins only when needed (events, tags, custom fields).
- Return:
  - count query
  - preview query (limit/offset, order by stable key)
  - export/paginated ids query for sending engine
- Handle edge cases:
  - empty groups
  - invalid operators for a field type
  - nulls
  - timezone-safe date filters
