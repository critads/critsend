# Critsend - Email Marketing Platform

## Overview

Critsend is an email marketing platform designed for efficient management and engagement of large subscriber bases. It offers comprehensive subscriber management with tag-based segmentation, a user-friendly campaign creation wizard, and flexible Mail Transfer Agent (MTA) configurations. The platform supports high-volume email sending with configurable speeds, detailed tracking (opens/clicks), and automatic unsubscribe handling. Key features include production-grade CSV import/export with batch processing, a complete REST API for integration, and capabilities to ensure email deliverability and data integrity. The business vision is to provide a scalable, reliable, and feature-rich solution for businesses to effectively communicate with their audience, driving engagement and marketing success.

## System Architecture

The Critsend platform uses a modern web stack: React, TypeScript, Vite, TailwindCSS, and Shadcn/UI for the frontend, and Express.js with TypeScript for the backend. PostgreSQL, managed with Drizzle ORM, serves as the primary data store, optimized for large datasets with advanced indexing. The system supports multi-user access with session-based authentication and AES-256-GCM encryption for sensitive data.

**UI/UX Decisions:**
The UI/UX follows Material Design 3 principles, featuring a clean, modern aesthetic with dark/light mode support, full mobile responsiveness, and a card-based layout. It uses Inter for text and JetBrains Mono for technical displays, with blue as the primary accent color.

**Technical Implementations & Design Choices:**
- **Authentication:** Session-based authentication with bcrypt for password hashing. All API routes require authentication, except public tracking, webhooks, health, and metrics endpoints.
- **Subscriber Management:** Features tag-based segmentation with GIN-indexed tags, a refs system for segment targeting, and a 30-day cooling-off period for unsubscribed contacts enforced by `suppressed_until` timestamps.
- **Campaign Management:** A 5-step wizard for campaign creation, including a WYSIWYG HTML editor and MTA selection. Supports A/B testing with variant tracking and automated winner declaration.
- **Email Sending:** High-performance sending via Nodemailer with configurable speeds and connection pooling. Supports concurrent campaign processing with a two-path architecture (in-memory nullsink and real SMTP).
- **Tracking:** Implements open tracking (1x1 pixel) and click tracking (redirects with open-redirect prevention). All tracking URLs are HMAC-SHA256 signed. Captures enriched context (IP, country, device, browser, OS) and uses a link registry for opaque UUID tokens in tracking links, along with branded short tracking URLs. Tracking events are buffered in-memory and flushed in batched multi-row inserts to a dedicated `pg.Pool` for isolation. **Split-transaction flush (Task #57 + #64):** for opens and clicks the `INSERT INTO campaign_stats` and the `UPDATE campaign_sends SET first_open_at/first_click_at` run as two separate transactions, each with `SET LOCAL lock_timeout = '2s'`. If the mark-firsts UPDATE fails (lock timeout), the raw events are still persisted and the 15-min counter-drift reconciler backfills the gap. **Coalesced counter bumps (Task #61):** engagement counter deltas are accumulated in-memory and flushed to the `campaigns` table every `TRACKING_COUNTER_COALESCE_MS` (default 8s) instead of per-flush-cycle (1.5s), reducing campaigns table write contention by ~5x. The 15-min counter-drift reconciler self-heals any gap. Tag enqueues (`storage.enqueueTagOperation`) remain post-commit and fire-and-forget. The flusher's "already counted?" pre-INSERT SELECT in `insertBatchAndBumpCounters` is backed by the `campaign_stats(campaign_id, subscriber_id, type)` covering index (bootstrap migration in `server/routes/tracking.ts`). **Cached engagement counters on `campaigns`:** `unique_opens_count`, `total_opens_count`, `unique_clicks_count`, `total_clicks_count`, `unsubscribes_count`, `complaints_count` — read by `/api/campaigns/stats` in a single fast SELECT instead of aggregating `campaign_stats` on every page load. **Counter-drift reconciler:** `server/workers/counter-reconciler.ts` runs every 15 min (configurable via `COUNTER_RECONCILE_INTERVAL_MS`) re-deriving `campaigns.sent_count`, `campaign_sends.first_open_at`, `first_click_at`, and the six engagement counters from the source-of-truth tables for campaigns active in the last `COUNTER_RECONCILE_WINDOW_HOURS=24` (full-table mode via `scripts/reconcile-counters.ts` for one-shot recovery). **Pool selection (Task #61):** the reconciler uses `effectivePool` — the tracking pool in the web process (where `TRACKING_POOL_MAX>0`) or the main pool in the worker process (where `TRACKING_POOL_MAX=0`) — so it actually works in both process types. Engagement reconciliation uses direct assignment from the source-of-truth `campaign_stats` aggregate, so it corrects both undercounts and overcounts (fixes inflated cached counters that caused >100% open rates on the /campaigns list). Metrics: `critsend_counter_drift_fixed_total{counter}` (labels: `sent_count`, `first_open_at`, `first_click_at`, `engagement_counters`), `critsend_counter_drift_run_duration_ms`, `critsend_counter_drift_last_run_timestamp_seconds` — non-zero `_fixed_total` indicates a regression in the live counter-write path. **Startup storm prevention (Task #61):** analytics rollup and engagement backfill are deferred by `WORKER_STARTUP_DELAY_MS` (default 30s) after worker/monolith startup. `runAnalyticsRollupSmart()` checks the most recent date in `analytics_daily` and only rolls up the gap (falls back to full 3650-day backfill only if the table is empty). **Follow-up audience index (Task #61):** `campaign_sends_campaign_first_open_idx` partial index on `(campaign_id, first_open_at) WHERE first_open_at IS NOT NULL` enables index scans for `getOpenersForParentCampaignCursor` / `countOpenersForParentCampaign`.
- **Segmentation (DSL v2):** Employs a recursive rule DSL for advanced segmentation with nested groups (AND/OR combinators, max 3 depth) and server-side pagination for segments. Includes targeted, cached subscriber counts with auto-invalidation.
- **CSV Import/Export:** Unified CSV format with auto-detection for refs and tags. Uses PostgreSQL COPY for high-performance imports with chunked file uploads, tag mode, pre-import operations, email deduplication, and live row counters. Supports forced tags/refs. **Remove Tags/Refs mode** allows uploading a CSV to strip specific tags/refs from matching subscribers without creating new ones — uses temp staging table with array overlap filtering (`&&`) for accurate update counts.
- **Job Queues:** Dual-mode system: PostgreSQL-backed queues with SKIP LOCKED and LISTEN/NOTIFY, and optional BullMQ + Redis for advanced features.
- **Automation Workflows:** Trigger-based email sequences with a multi-step workflow builder.
- **Advanced Analytics:** Comprehensive analytics including engagement trends, cohort analysis, deliverability, and subscriber growth.
- **Prometheus Metrics:** Full observability via a `/metrics` endpoint covering key platform operations. An in-app **System Metrics** dashboard (`/system-metrics`) reads from `/api/system-metrics` (JSON projection of the Prometheus registry) and displays 5xx errors, load-shed counts, pool health, tracking/bounce buffer stats, counter-drift, and system info with 10s auto-refresh.
- **Security:** CSRF protection, Helmet.js security headers with CSP, CORS, extensive input validation, HTML sanitization, 5-tier rate limiting, secure session management, and webhook authentication.
- **Robustness:** Includes graceful shutdown, memory monitoring with load shedding, automated campaign auto-resume, bounce webhook idempotency, and bulk-optimized batch webhook processing. Web-process guardians rescue stuck import and campaign jobs. **Bootstrap migration safety (Task #64):** `server/bootstrap-lock.ts` provides PostgreSQL advisory lock guards (`pg_try_advisory_lock`) and fast `pg_indexes` pre-checks for all startup bootstrap migrations (tracking tokens, tracking routes, analytics indexes, trigram index). Only one process (web or worker) runs each migration group; the other skips. All index builds on large tables use `CREATE INDEX CONCURRENTLY` to avoid blocking writes. The `suppressed_until` backfill is a one-time operation (skipped once data exists).
- **Repository Pattern:** Storage layer decomposed into focused repository modules for subscriber, campaign, import, MTA, job, and system management.
- **Data Integrity & Concurrency:** Utilizes PostgreSQL's transactional capabilities, atomic counter updates, unique indexes, and optimistic locking.
- **Real-Time SSE Progress:** Server-Sent Events push instant progress updates for import, flush, and campaign jobs, with Redis integration for split-process mode.
- **Process Separation:** Supports a split-process architecture with dedicated web server (main pool 14 + tracking pool 10) and worker (`WORKER_PG_POOL_MAX=18`, `MAX_CONCURRENT_CAMPAIGNS=12`) processes. Total budget 48/50 with headroom 2.
- **Pool Saturation Safety Net:** Layered defenses guarantee users never see a 500 from pool starvation. All 503 responses use the strict body `{ error: "service_busy" }` + `Retry-After: 1` header (no extra fields, so strict-equality client checks are stable). (a) `connectionTimeoutMillis=2s` so checkout failures surface fast. (b) `loadShedMiddleware` rejects non-critical requests with 503 when `pool.waitingCount>0` OR saturation>0.9; `poolErrorHandler` converts pg checkout timeouts to 503. Critical paths bypass: `/api/health`, `/metrics`, `/api/track/*`, `/c/*`, `/u/*`, `/api/unsubscribe/*`, `/api/webhooks/*`, `/api/auth/*`. (c) `requestLeaseMiddleware` + `installRequestLeaseTracker` use AsyncLocalStorage and a `pool.connect` monkey-patch (handles both Promise and callback forms) to enforce `MAX_CONNECTIONS_PER_REQUEST=2`. Per-route gauge `critsend_db_pool_request_holding{route}` exposes live lease counts. (d) Bounce webhooks buffer in `server/bounce-buffer.ts` (1.5 s flush, dedupe by `(email|messageId)` with `(email|type)` fallback, 60 s window, capped 25 k) and flush via the shared tracking pool — Mailgun/SES retry storms never touch the main pool. Webhook handlers respond 202 immediately. (e) `campaign-sender.ts` awaits previous batch's `flushPromise` before starting next prefetch — each campaign holds ≤1 main-pool conn at a time. (f) Metrics: `critsend_db_pool_load_shed_total{reason}`, `critsend_db_pool_checkout_timeout_total`, `critsend_db_pool_request_holding{route}`, `critsend_db_pool_request_lease_exceeded_total{route}`, `critsend_bounce_buffer_*`. Alert rules: `docs/prometheus-alerts.md`. Load test: `scripts/load-test-pool-saturation.sh` automates 12 nullsink campaigns + bounce/pixel firehose and asserts zero 5xx, every 503 carries Retry-After, `/api/health` p99 < 500 ms. (g) **Tracking-pool 503 conversion (Task #57):** `getLinkDestinationCached` retries once after 100 ms on a tracking-pool checkout timeout; if both attempts fail it throws a typed `TrackingPoolUnavailableError` which the click routes (`/c/:token`, `/api/track/click/:cid/:sid?lid=…`) translate into the canonical `503 + Retry-After: 1 + {error:"service_busy"}` response instead of the legacy generic `500 "Tracking error"`. Necessary because `markPoolErrorOnRequest` is only wired to the main pool's `pool.connect` monkey-patch — tracking-pool checkout errors don't auto-upgrade through `poolErrorResponseUpgrade`, so the click route has to do the translation explicitly.
- **Nullsink SMTP Testing:** An internal SMTP server allows for testing campaigns without sending real emails.
- **Self-Hosted Deployment:** Provided `deploy/` directory includes PM2 configuration, Nginx setup, idempotent bootstrap script, deployment script, and GitHub Actions workflow for automated deployments.
