# Critsend - Email Marketing Platform

## Overview

Critsend is a robust email marketing platform designed to manage and engage large subscriber bases efficiently. It offers comprehensive subscriber management with advanced tag-based segmentation, a user-friendly campaign creation wizard with WYSIWYG editing, and flexible Mail Transfer Agent (MTA) configurations. The platform supports high-volume email sending with configurable speeds, detailed email tracking (opens/clicks), and automatic unsubscribe handling. Key capabilities include production-grade CSV import/export with batch processing, a complete REST API for integration, and features to ensure email deliverability and data integrity. The business vision is to provide a scalable, reliable, and feature-rich solution for businesses to effectively communicate with their audience, driving engagement and marketing success.

## System Architecture

The Critsend platform is built with a modern web stack, featuring a React, TypeScript, Vite, TailwindCSS, and Shadcn/UI frontend for a responsive and intuitive user experience. The backend is an Express.js and TypeScript application, ensuring robust API services. PostgreSQL, coupled with Drizzle ORM, serves as the primary data store, optimized for handling millions of records with advanced indexing for efficient querying.

**UI/UX Decisions:**
The UI/UX adheres to Material Design 3 principles, offering a clean, modern aesthetic with dark/light mode support and full mobile responsiveness. It utilizes Inter for primary text and JetBrains Mono for technical displays, with blue as the primary accent color. A card-based layout organizes information effectively.

**Technical Implementations & Design Choices:**
- **Authentication:** Session-based auth with bcrypt password hashing (cost factor 12). Open registration for multi-user access. All API routes protected by auth middleware except public endpoints (tracking, webhooks, health, metrics). Login/register/logout endpoints at `/api/auth/*`.
- **Subscriber Management:** Tag-based segmentation with GIN-indexed tags for fast filtering.
- **Campaign Management:** A 6-step wizard guides campaign creation with WYSIWYG HTML editing.
- **Email Sending:** High-performance campaign sending with configurable speeds (500-60000 emails/min) via Nodemailer with connection pooling. Godzilla mode: 250 concurrent workers, 60k/min rate cap, pooled nullsink transporter (200 maxConnections). Bulk send recording batches DB operations (bulkReserveSendSlots + bulkFinalizeSends) to reduce per-email overhead from 2 DB round-trips to ~2 per batch. Intermediate flush every 500 emails keeps sentCount accurate in near-real-time. Bulk nullsink capture inserts (bulkCreateNullsinkCaptures) replace per-email inserts. Fire-and-forget error logging avoids blocking the send pipeline. Campaign job heartbeat prevents stale cleanup during long sends. SMTP maxConnections scales per-MTA (default 10, configurable via maxSmtpConnections). LISTEN/NOTIFY triggers instant job pickup. Cached status checks (10s TTL) via lightweight getCampaignStatus query. Automatic fallback to individual operations on bulk failure.
- **Sending Engine V3 (Feb 2026):** Two-path architecture: Path A (nullsink) bypasses SMTP entirely via `sendEmailBatchNullsink()` — processes emails synchronously in-memory with pre-computed base HTML template, no TCP/SMTP overhead. Path B (real SMTP) uses chunked concurrency with pooled transporters. Write-behind buffer accumulates results and flushes to DB on 2500/5000-count or 3/5-second thresholds. Batch size 10k (SMTP) / 15k (nullsink) subscribers per DB fetch. Sub-batching at 2500 for nullsink. Pipeline prefetch overlaps next DB fetch with current batch processing. Async flush fires DB writes without blocking the send loop. bulkReserveSendSlots uses UNNEST for efficient bulk inserts. bulkFinalizeSends uses CTEs to combine success/failed/counter updates into single DB round-trip. Flush operations parallelized via Promise.all (finalize + nullsink captures). Speed configs: medium 2000/min@30 concurrent, fast 5000/min@80, godzilla 60k/min@250.
- **Tracking:** Open tracking (1x1 pixel), click tracking (redirects with open-redirect prevention), and optional tagging on interactions. All tracking URLs are HMAC-SHA256 signed with timing-safe verification. Click tracker validates URL protocols (http/https only) and rejects invalid signatures with 403.
- **Segmentation:** Rule-based filtering with support for nested groups, various field operators, and performance optimizations like segment count caching and specialized database indexes (pg_trgm for email, GIN for tags).
- **CSV Import/Export:** High-performance import engine using PostgreSQL COPY (pg-copy-streams) with concurrent batch pipeline (4 parallel COPY operations, 25k rows per batch). COPY streams data into temp staging tables, then merges via INSERT...ON CONFLICT with tag merge/override support. Semaphore-based backpressure controls memory usage. Fallback to INSERT-based upserts if COPY fails. Chunked file uploads supporting up to 1GB files. CSV-only file type validation. Exports utilize streaming for large datasets.
- **Job Queues:** PostgreSQL-backed job queues (`campaign_jobs`, `import_job_queue`) with `FOR UPDATE SKIP LOCKED` ensure race-condition-free multi-worker processing and crash recovery for campaign sending and CSV imports. Enhanced with PostgreSQL LISTEN/NOTIFY (`server/message-queue.ts`) for near-instant job pickup with advisory lock helpers and automatic fallback to polling on LISTEN disconnection.
- **Reliable Tag Queue:** Asynchronous, guaranteed delivery of tag operations through a `pending_tag_operations` table with retry logic and a background worker.
- **Subscriber Flush System:** Background job for large subscriber deletions with progress tracking and cancellation.
- **A/B Testing:** Campaign variant management with split allocation, per-variant tracking (opens/clicks/bounces), statistical significance calculation (proportion z-test with 90/95/99% confidence levels), and winner declaration that updates the campaign. Routes at `/api/campaigns/:id/ab-test/*`.
- **IP Warmup Schedules:** Automated warmup for new MTA IPs/domains with configurable ramp curves (exponential multiplier), daily volume caps, pause/resume controls, and progress tracking. Routes at `/api/warmup/*`.
- **Automation Workflows:** Trigger-based email sequences (subscriber_added, tag_added, tag_removed, subscriber_opened, subscriber_clicked) with multi-step workflow builder supporting send_email, wait, add_tag, remove_tag steps. Enrollment management with deduplication. Routes at `/api/automation/*`.
- **Advanced Analytics:** Comprehensive analytics with engagement trends, cohort analysis (weekly/monthly), deliverability metrics (per-MTA breakdown), subscriber growth tracking, top campaign rankings, and manual rollup aggregation. Routes at `/api/analytics/*`.
- **Prometheus Metrics:** Full observability via `/metrics` endpoint (prom-client) with counters/gauges/histograms for emails sent, queue depth, job processing duration, HTTP requests, DB pool stats, memory usage, active campaigns, and automation enrollments. Background collector updates every 15s.
- **MTA Password Encryption:** AES-256-GCM encryption at rest for SMTP credentials, backwards-compatible with legacy plaintext. Passwords masked in list views.
- **Database Health & Maintenance:** Automated database cleanup system with configurable retention rules per table. Background worker runs every 6 hours with batched DELETE operations (1000 rows/batch, 50k max per run). 7 default rules covering nullsink_captures (7d), campaign_sends (180d), pending_tag_operations (7d), campaign_jobs (30d), import_job_queue (30d), error_logs (30d), session (7d). Manual trigger via API/UI. Mutex prevents concurrent runs. Full cleanup history logging. Routes at `/api/database-health/*`.
- **Security:** CSRF protection with timing-safe validation, Helmet.js security headers with CSP configured, CORS middleware with configurable origins, extensive input validation, HTML sanitization, 5-tier rate limiting, secure session management (httpOnly, sameSite=lax, 24h lifetime), webhook authentication with timing-safe X-Webhook-Secret, error response sanitization, SESSION_SECRET enforcement in production.
- **Email Send Retry (12h Window):** Two-tier retry system. Tier 1 (individual): After main send loop, failed emails are retried with exponential backoff (30s→1m→2m→...15m cap) within a 12-hour window from campaign start. Failed sends marked for retry (status reset to pending, retry_count incremented) and re-sent via same SMTP/nullsink path. Tier 2 (campaign-level): When entire campaign fails or pauses (MTA down, fatal error), the job is auto-re-enqueued with delayed retry (30s→1m→2m→...15m cap backoff) if within 12h window. Campaign status reset to sending, new job created with next_retry_at delay. Both tiers respect campaign pause/cancel signals and maintain heartbeats during backoff waits.
- **Robustness:** Graceful shutdown with connection draining, memory monitoring with load shedding (503 responses when critical), automated campaign auto-resume on server restart, bounce webhook idempotency via tag-based deduplication.
- **Audit Hardening (Feb 2026):** Auth rate limiter (10 attempts/15min), request correlation IDs (x-request-id), CSV export formula injection prevention, response body logging redaction, deleteAllSubscribers OOM fix (SQL rowCount), DB transactions for campaign create/update/resume+job enqueue and import job creation+queue, MTA transporter pool invalidation on update/delete, LIKE wildcard escaping in segment builder, TRACKING_SECRET lazy initialization, DB health check in /api/health, campaign rate limiter applied to routes, TypeScript LSP errors resolved (logger accepts unknown types).

**System Design Choices:**
- **Modular Route Architecture (Feb 2026 Refactor):** Fully modular architecture. `server/routes.ts` is a thin 100-line orchestrator that sets up rate limiters and registers 14 route modules under `server/routes/` (subscribers, segments, mtas, tracking, webhooks, analytics, ab-testing, warmup, automation, advanced-analytics, campaigns, import-export, health, nullsink). Shared utilities extracted to `server/utils.ts`, multer/upload configs to `server/upload.ts`, campaign sending engine to `server/services/campaign-sender.ts`, all background workers to `server/workers.ts`. Storage organized with 16 section headers.
- **Data Integrity & Concurrency:** Utilizes PostgreSQL's transactional capabilities, atomic counter updates, unique indexes, and optimistic locking to prevent race conditions and ensure data consistency during high-volume operations like email sending.
- **Production Architecture:** Employs PostgreSQL-backed job queues for horizontal scaling and crash recovery of background processes. Chunked file upload and persistent object storage handle large file operations efficiently.
- **Nullsink SMTP Testing:** An internal SMTP server allows for testing campaigns without sending real emails, offering configurable latency and failure injection for robust testing.
- **Database Migrations:** Uses Drizzle Kit generate for migration files (in `./migrations/`) providing an audit trail of schema changes.
- **Testing:** Vitest test framework with unit tests covering crypto module, tracking URL signing, schema validation, and logger. 17 tests in 4 test files.

## External Dependencies

- **PostgreSQL (Neon):** Primary database hosted on Neon (external managed PostgreSQL). Connection via `NEON_DATABASE_URL` secret with SSL and `search_path=public` set per-connection. Falls back to `DATABASE_URL` if Neon URL is not set. Pool and LISTEN/NOTIFY client both auto-detect Neon and configure SSL. The Neon pooler endpoint is used for connection pooling.
- **Nodemailer:** Used for real SMTP email sending with connection pooling and retry mechanisms.
- **`sanitize-html`:** For sanitizing HTML content in campaigns to prevent injection attacks.
- **`connect-pg-simple`:** For persistent session management using PostgreSQL.
- **`bcrypt`:** For secure password hashing with configurable cost factor.
- **`helmet`:** For HTTP security headers (X-Content-Type-Options, X-Frame-Options, etc.).
- **`prom-client`:** Prometheus metrics client for Node.js, providing counters, gauges, histograms, and default process metrics.
- **Replit App Storage:** Utilized for persistent object storage of CSV files, ensuring data survives deployments.
